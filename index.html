About Pint

Dependency Aware

Your project's build tasks form a directed acyclic graph which Pint uses, like many other mature build tools to create a safe build sequence.

Concurrent

Because Pint is dependency aware it will safely and transparently parallelize your build by spawning multiple Grunt processes.

Uses Grunt and Grunt plugins

Pint runs Grunt processes under the hood so you can continue to take advantage of the Grunt plugin ecosystem and can reuse all your existing Grunt config.

Problems Pint is Trying to Solve

Grunt ushered in an era of sophisticated frontend build process with by lowering the engineering cost of good tooling. As the Grunt bounded towards near ubiquity in a modern frontend stack, a typical build has gone from a couple of small in house scripts to a Gruntfile accompanied by a half dozen of more of the plugins available from the vibrant developer community.

A typical Gruntfile starts with, at a minimum some variation of: jsHint, jasmine, LESS, handlebars, uglify, copy and clean stack. Just these half dozen or so plugins can baloon your Gruntfile upwards of 300 lines and when you add complex concatenation, cache busting, and versioning can cause it to grow well into the 1000+ line range. At this point some of the problems with stock grunt begin to become apparent.

  • Tasks are grouped by plugin not rather than by the job they are doing. For example I will frequently have a copy task to move my styles, JavaScript and static files at different points in my build process. Rather than configuring them with the rest of the associated Style, JavaScript or static build steps I just configure them with all the other things that get copied.

  • Grunt provides no way to declaratively enumerate dependencies. Uglify may depend on Handlebars compilation and Copy may depend on Less which all depends on passing tests and jsHint. The only way to capture these dependencies is to register very specific tasks the result of this is increased cognitive overhead for your development team.

  • No concurrency. Development machines are powerful computers not subject to the constraints of the browser, a good build tool should take advantage of concurrency.

  • Common development features like versioning and environment profiles are not handled well. There are a number of somewhat universal tasks that developers are usually left to cobble together themselves. For example most often you will not to develop with minified code so you will have a development task and a production task.

The Pint Mission

There is more to build than tasks

  • Build should be concurrent by default
  • Build should manage dependencies transparently
  • Build should be well organized
  • Build tasks should be independant from one another

Getting Started

Install Pint


$ npm install -g pint
$ pt --help

Add Pint to your project


{
  "name": "my-project-name",
  "version": "0.1.0",
  "devDependencies": {
    "pint": "0.1.0",
    "grunt-contrib-jshint": "0.8.0"
  }
}
./package.json

Create a new pint task

Notice that the config here is exactly what you would put in your Gruntfile.js and the build array is just a list of what would be Grunt tasks. 


'use strict';

module.exports = {
  name: 'test',

  dependsOn: [],

  build: [
    'jshint',
  ],

  config: {
    jshint: {
      options: {
        ignores: ['./node_modules/**/*']
      },
      all: ['./**/*.js']
    }
  }
};
./build/test.js

Create a Pint file and require in your build files


'use strict';

module.exports = {
  jobs: [
    require('./build/test.js'),
  ]
};
./Pint.js

Install packages and run Pint


$ npm install
$ pt
